Purpose

- This document is Stage 3 from `docs/Prompt Forge — Lane Taxonomy Migration Plan.txt`.
- The goal is to identify the safest future readers of taxonomy metadata before any implementation is considered.
- This is a consumer inventory only. It does not propose code edits or runtime behavior changes.


Inventory categories

- `Safe first reader`
  - a read-only grouping, summary, or presentation surface that could consume category/profile metadata without changing routing, resolver precedence, persistence, or orchestration timing
- `Possible later reader`
  - a surface where taxonomy metadata might help later, but only after a first read-only use proves stable
- `True leaf-only concern`
  - a surface that still fundamentally depends on exact lane identity rather than category/profile classification
- `Unsafe first reader`
  - a timing-sensitive, precedence-owning, or behavior-owning surface where taxonomy should not be introduced first


Consumer inventory table

  - justification: picker organization is the cleanest user-facing grouping surface and can remain read-only if taxonomy is only used to group or present existing intent names

Best first consumer candidate

- safest first future reader: `PromptForge.App/ViewModels/MainWindowViewModel.cs` intent picker / grouping presentation surface
- why this is safest:
  - it is read-only presentation
  - it can consume `Category` without changing resolver routing, builder routing, fallback behavior, or persistence semantics
  - it tests whether taxonomy adds real user-facing value before any central behavior surface is touched
  - it is safer than `IntentModeSummary` because picker grouping can stay purely organizational, while summaries already mix in more special-case wording


Unsafe first consumers

- `PromptForge.Core/Services/SliderLanguageCatalog.cs` dispatch and fallback surfaces
  - most dangerous because they own resolver precedence and fallback behavior
- `PromptForge.Core/Services/PromptBuilderService.cs` build entry and standard-path routing surfaces
  - dangerous because they own branch selection and final builder timing
- `PromptForge.App/ViewModels/MainWindowViewModel.cs` stabilized orchestration shells
  - dangerous because they own timing-sensitive UI behavior, access gating, apply/reset sequencing, and regeneration timing
- `StandardLaneStateAdapter` and config replay surfaces
  - dangerous because taxonomy does not belong in hydration or persistence paths first


What taxonomy will not solve

- exact unlock behavior for `Vintage Bend`, `Product Photography`, `Food Photography`, and `Lifestyle Advertising Photography`
- exact contributor ownership in `LanePromptContributorRegistry`
- exact lane-local phrase authority in `SliderLanguageCatalog.[Lane].cs`
- exact guide-text routing in `MainWindowViewModel`
- exact config capture/apply/reset fields for each lane leaf
- exact builder branch ownership where a lane has special descriptor or contributor behavior
- exact resolver dispatch and fallback precedence

Why not

- these are still leaf-owned or behavior-owned concerns
- taxonomy can classify them, but classification does not replace exact lane identity where the runtime still needs the exact leaf


Assessment

- Yes, the consumer inventory supports a future minimal read-only implementation phase.
- The strongest evidence is that at least one genuinely safe first reader exists:
  - picker / grouping presentation in `MainWindowViewModel`
- The inventory also shows clear boundaries where taxonomy should not be introduced first, which makes the later implementation phase easier to scope honestly.


Recommendation

- `Proceed to Stage 4 later`
- The inventory shows one clear low-risk first reader, several possible later readers, and a large set of true leaf-only or unsafe behavior surfaces, which is exactly the kind of evidence needed before considering a minimal read-only implementation phase.
