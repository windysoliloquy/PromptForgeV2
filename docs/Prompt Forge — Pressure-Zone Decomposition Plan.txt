Approval thresholds for future moves

Any future containment or extraction move inside the pressure zones should require all of the following:

Call-order equivalence
- The proposal must identify the exact caller sites affected.
- It must prove the moved logic runs in the same order relative to surrounding work.

Notification timing
- Any `OnPropertyChanged(...)` batch must remain identical in membership and order unless an explicit behavior change is approved.
- Any command-state refresh must remain at the same timing boundary.

Regeneration timing
- Any `RegeneratePrompt()` call site must stay in the same location and under the same guard conditions unless an explicit behavior change is approved.

Compression / cleanup timing
- Any move touching `PromptBuilderService` must preserve when compression runs, when cleanup runs, and what fragment set reaches each phase.

Resolver precedence
- Any move touching `SliderLanguageCatalog` shared routing must prove that the same lane resolver or fallback path wins for the same configuration.

Persistence semantics
- Any move touching configuration capture/apply/reset or preset flow must preserve the same `PromptConfiguration` contents, replay order, and save/load/reset timing.

Unchanged wording behavior
- No move is acceptable if it changes emitted phrase text, fallback wording, or preserve-from-compression behavior unless that wording change is the explicit goal of the work.

Scope proof
- The proposed move must name the exact members involved.
- It must state what remains in place.
- It must avoid widening into nearby orchestration clusters unless separately approved.

Verification proof
- At minimum: successful `dotnet build PromptForge.sln --no-restore`
- Plus a targeted code-review equivalence check against:
  - call-order behavior
  - notification timing
  - regeneration timing
  - compression/cleanup timing
  - resolver precedence
  - persistence semantics
  - unchanged wording behavior
